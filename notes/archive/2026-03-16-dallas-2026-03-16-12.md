### Pattern
1. GET PR threads via ADO REST API
2. Check for APPROVE verdicts and existing fix commits
3. If all issues resolved, post closed-status thread (status: 4) and exit
4. Total time: ~15 seconds vs 5-10 minutes for full worktree + yarn install + build + test + lint

### Recommendation for Engine
The dispatch system needs to:
1. Check existing reviewer votes before dispatching review-fix tasks
2. Distinguish implementation notes from actionable code review feedback
3. Check for prior "No Action Required" threads to avoid infinite re-dispatch loops

```skill
# PR Review Fix — Early Bail-Out

## Steps
1. Get ADO token: `az account get-access-token --resource 499b84ac-1321-427f-aa17-267ca6975798 --query accessToken -o tsv`
2. Fetch PR threads: `GET https://dev.azure.com/{org}/{project}/_apis/git/repositories/{repoId}/pullRequests/{prId}/threads?api-version=7.1`
3. Check for:
   - APPROVE verdicts (threads containing "APPROVE" or "Verdict: APPROVE")
   - Existing fix commits (threads mentioning "fix:" or "Review Fixes Applied")
   - Prior "No Action Required" threads (status: 4/closed)
   - Reviewer votes of 10 (approve)
4. If all issues resolved: post a closed-status (status: 4) thread confirming no action needed, and exit
5. If actionable issues remain: proceed with worktree setup and fixes

## Notes
- The "review findings" may actually be implementation notes, not actionable issues
- Check thread status: 4 = closed/resolved, 1 = active, undefined = system-generated
- Use dev.azure.com hostname (not office.visualstudio.com which returns 302)
- Write JSON payloads to $TEMP/file.json on Windows (not /dev/stdin)
```
