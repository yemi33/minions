## Architecture

### High-Level Overview
OfficeAgent is an AI-powered document generation platform running in Docker containers. It uses Claude SDK and Azure OpenAI (via GitHub Copilot SDK) to create DOCX, PPTX, XLSX, and PDF files. The system is a Yarn 4.10.3 monorepo with lage task orchestration, TypeScript 5.9, Node 24+, and a Python evaluation framework.

### Message Flow
```
Client (WebSocket on :6010)
  Ôćĺ API Module (app.ts) Ôćĺ WebSocket/HTTP routing
    Ôćĺ Core (WebSocketManager/Router)
      Ôćĺ Handler Registry (typed message dispatch)
        Ôćĺ Orchestrator (Claude SDK or GHCP)
          Ôćĺ Grounding Pipeline (enterprise search, image, web)
            Ôćĺ Agent execution (skills, sub-agents, hooks)
              Ôćĺ Chain-of-Thought tracking
                Ôćĺ Response streamed back to client
```

### Three-Tier Agent Architecture
1. **Full Claude SDK agents** (create-agent): Multi-version, sub-agents, 20+ flights, request interceptors, hooks by agent type. The most complex ÔÇö `create-agent.ts` is ~84KB.
2. **Copilot SDK agents** (office-agent, excel-agent): Simpler, use GhcpOrchestrator, import shared services from create-agent.
3. **Minimal agents** (ppt-creation-agent): No LLM, direct HTMLÔćĺPPTX conversion, bare registry.

### Orchestrator Abstraction
`@officeagent/orchestrator` provides a provider-agnostic `IOrchestrator` interface:
- `ClaudeOrchestrator` ÔÇö Claude SDK wrapper
- `GhcpOrchestrator` ÔÇö GitHub Copilot SDK wrapper
- Factory pattern: `createOrchestrator()` selects provider at runtime
- Common types: `OrchestratorRequest`, `OrchestratorMessage`, `OrchestratorResult`

## Patterns

### 1. Registry-First Agent Design
Every agent exports `agentRegistry: AgentRegistryConfig` from `src/registry.ts`:
- Identity: id, name, packageName, description
- Versions array with model, tools, skills, subagents, scripts
- Flights (static feature flags) and Filters (dynamic runtime rules)
- Default version selection + flight-based version overrides

### 2. Versioned Prompt Organization
```
src/prompts/<version>/
ÔöťÔöÇÔöÇ system-prompt.md (EJS templates with <%- variable %>)
ÔöťÔöÇÔöÇ rai.md
ÔöťÔöÇÔöÇ skills/<name>/SKILL.md (YAML frontmatter + instructions)
ÔöťÔöÇÔöÇ subagents/<name>.md (YAML frontmatter: name, description, tools)
ÔöťÔöÇÔöÇ scripts/ (JS/Python runtime scripts, 30+ in create-agent v1.1.0)
ÔöťÔöÇÔöÇ agent-capabilities/ (per-doc-type: word.md, excel.md, powerpoint.md)
ÔöťÔöÇÔöÇ intent-detection/ (CLARIFICATION.md, CHAT_REPLY.md, etc.)
ÔööÔöÇÔöÇ overwrites/ (flight-conditional file replacements)
```

### 3. Skills vs Sub-Agents
- **Skills**: Instructions + scripts the agent follows directly (shared context window)
- **Sub-agents**: Isolated Claude instances with own system prompt (parallel, focused)
- Pattern: well-scoped pipeline steps Ôćĺ sub-agent; general capabilities Ôćĺ skill
- **Create-agent sub-agent pipeline example**:
  - DOCX: `@docx-outline-generator` Ôćĺ parallel `@docx-section-generator` instances
  - PPTX: `@pptx-outline-generator` Ôćĺ `@pptx-html-creator-validator` Ôćĺ `@pptx-html-updater`
  - XLSX: `@xlsx-resource-discovery` Ôćĺ `@xlsx-content-extractor` Ôćĺ `@xlsx-outline-generator` Ôćĺ `@xlsx-office-js-generator`

### 4. Hook System
- `HookRegistry` with per-agent-type providers (Base, Word, Excel, PowerPoint)
- Hooks: PreToolUse, PostToolUse, SessionStart, SessionEnd, startTurn
- Used for performance tracking, CoT streaming, output validation

### 5. Flight System (Feature Gating)
- Feature flags in registry, enabled via `OAGENT_FLIGHTS=<key>:true`
- Can override: version, model, subagents, scripts, raiGuidelines, entire prompt files
- Flights scoped to specific versions via `applicableVersions`
- Flight overwrite mechanism: `{ src: 'overwrites/file.md', dst: 'skills/path/file.md' }`
- Example flights in create-agent: `@1.1.0`, `@1.2.0`, Researcher, AutoThemeGeneration, NewDesign, DocxAutoMode

### 6. Grounding Pipeline (4-Stage)
`@officeagent/grounding` implements: Preprocessor Ôćĺ Retriever Ôćĺ Postprocessor Ôćĺ Storage
- Entity-specific extractors: Email, File, Chat, Meeting, People, Calendar, Transcript
- A2A (Agent-to-Agent) HTTP communication pattern

### 7. Internal Handler Registry
API module provides typed handlers for internal (non-WebSocket) operations:
- WebSearchHandler, EnterpriseSearchHandler, EnterpriseFetchHandler
- UserContextHandler, ImageSearchHandler, CitationInjectionHandler
- ExcelParseHandler, PptxSlideCountHandler
- Supports both streaming (SSE) and synchronous responses

### 8. Build & Runtime
- PowerShell required for yarn/oagent/gulp commands on Windows
- Docker multi-stage build: poppler + LibreOffice 26.2 + Node 24 + OfficePy base image
- Ports: 6010 (external API), 6011 (internal), 6020 (debug), 7010 (grounding)
- Hot reload: `yarn reload` copies dist to running container
- Health check: `curl http://localhost:6010/health`

## Recommendations

1. **For new agent work**: Start from `agents/create-agent/src/registry.ts` as the canonical reference. Read the agent's CLAUDE.md if present.
2. **For prompt changes**: Use the flight-based overwrite system rather than mutating base version files. Enables safe A/B testing.
3. **For shared module changes**: Run `yarn build` (not just compile) to catch Docker packaging issues early.
4. **For exploration tasks**: `oagent gen "prompt" --open` is the fastest feedback loop.
5. **For eval work**: Activate the Python venv first (`.\venv\Scripts\Activate.ps1`), then use `oagent eval alias` to discover available test suites.
