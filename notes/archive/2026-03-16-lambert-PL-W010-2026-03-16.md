## Patterns Discovered

### 1. office-bohemia Package Creation Pattern
To add a new package to office-bohemia: (source: `packages/copilot/` as reference)
1. Create `packages/<name>/package.json` with `@fluidx/<name>` scope, `private: true`, `version: "0.0.1"`
2. Create `tsconfig.json` extending `../../tsconfig.json` with `outDir: "lib/"`, `rootDir: "src"`
3. Create `tsconfig.typecheck.json` extending `./tsconfig.json` with `composite: false`, `emitDeclarationOnly: true`
4. Create `just.config.cjs` requiring `@fluidx/office-bohemia-build-tools/just-preset` and calling `preset()`
5. Run `yarn install` (NOT `--mode update-lockfile` alone — need full install for node_modules state)
6. Build with `yarn workspace @fluidx/<name> build`

### 2. SharedTree Schema Pattern in office-bohemia
(source: `packages/copilot/src/model/sharedTree/CopilotHistoryTreeSchema.ts`, `packages/conversa/src/schema/SharedTreeSchema.ts`)

```typescript
const sf = new SchemaFactoryBeta('com.microsoft.loopcomponent.<component>');
// All node types MUST use { allowUnknownOptionalFields: true } for forward-compat
export class MyNode extends sf.object('MyNode', { ... }, { allowUnknownOptionalFields: true }) {}
export const treeConfiguration = new TreeViewConfiguration({ schema: RootSchema });
```

- `SchemaFactoryBeta` from `@fluidframework/tree/beta` (not plain `SchemaFactory`)
- Namespace convention: `com.microsoft.loopcomponent.<component-name>`
- `@fluidframework/tree` version range: `>=2.90.0 <2.100.0`
- `@fluidx/fluid-extension` v8.21.1 provides `TreeDataObject`, `makeTreeDataModelAdapter()`

### 3. Tree → Jotai Bridge Pattern (new pattern established)
(source: `packages/cowork-component/src/sharedTree/coworkTreeAdapter.ts`)

For Bebop features using Jotai (not MobX like conversa):
1. Create adapter class with `EventEmitter`
2. Subscribe to `Tree.on(node, 'treeChanged', callback)` for each subtree
3. Project tree nodes to plain readonly JS snapshot objects
4. Emit snapshots via `stateChange` event
5. In Jotai, create `syncFromSnapshotAtom` write-atom that accepts snapshot
6. In hook, call `adapter.onStateChange(snapshot => set(syncFromSnapshotAtom, snapshot))`

This decouples Jotai atoms from SharedTree completely — UI only depends on atoms.

### 4. yarn install in worktrees
(source: build failure during PL-W010 implementation)

When adding a new package to office-bohemia in a worktree:
- `yarn install --mode update-lockfile` updates the lockfile but does NOT create `node_modules` state
- Must run full `yarn install` (~14min) to get buildable state
- The lockfile is shared across worktrees (via git), but node_modules state is per-worktree

## Gotchas

- **tsconfig.typecheck.json is mandatory**: Build pipeline (`lage`) expects this file separately from `tsconfig.json`. Without it, typecheck task fails immediately. (source: build error during PL-W010)

- **Unused imports are TS errors**: The typecheck config treats unused imports as errors (TS6196). Clean up imports before committing. (source: build failure with ChatMessageNode, ArtifactNode, ParticipantNode unused)

- **office-bohemia yarn install takes ~14 minutes**: Plan for this when creating new packages in worktrees. The 227-package monorepo resolution is slow.

- **SharedTree `for...of` on map**: `root.participants` (SharedTree map) supports `for (const [key, value] of map)` iteration, but the typing may require explicit casting in some TS versions.

- **No barrel files in Bebop features but YES in packages**: `packages/cowork-component/src/index.ts` is a legitimate package entry point. The no-barrel-files rule only applies to `apps/bebop/src/features/**`.

## Schema Design Rationale

The CoworkSchema has 6 top-level fields:
- `session` (SessionMetadataNode) — 1:1, always present
- `participants` (Map<ParticipantNode>) — keyed by userId for O(1) lookup
- `messages` (ChatMessageList array) — ordered, append-only in practice
- `progressionSteps` (ProgressionStepList array) — ordered, upsertable by id
- `artifacts` (ArtifactList array) — ordered, append-only
- `arbitraryProperties` (ArbitraryProperties) — extensible metadata bag with schemaVersion

All status/role/type fields use `SchemaFactory.string` (not enums) so new values can be added without schema migration. TypeScript string union types provide compile-time safety at the consumer layer.
