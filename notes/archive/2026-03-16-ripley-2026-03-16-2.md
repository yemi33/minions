## Verdict: REQUEST_CHANGES

### Critical Finding: Wire Format Field Name Mismatches

**FileInfo mirror is completely wrong.** The PR's mirrored `FileInfo` interface uses field names (`fileId`, `fileName`, `fileType`, `fileUrl`, `fileSize`, `createdAt`) that do NOT exist in OfficeAgent's actual `FileInfo` interface. The real fields are `path`, `filename`, `size`, `content`, `isEdit`, `encoding`, `driveId`, `driveItemId`.
(source: `modules/message-protocol/src/types/core.ts:297-314` in OfficeAgent repo)

**QueryStatusPayload mirror has wrong field names.** PR uses `errorMessage` (real: `error`), `slideCount` as number (real: `slideCountInfo` as object), and adds nonexistent fields `message` and `progress`.
(source: `modules/message-protocol/src/types/core.ts:360-371` in OfficeAgent repo)

**Impact:** JSON deserialization from WebSocket messages will silently produce `undefined` for all mismatched fields. The adapter's `extractArtifact()` and `mapQueryStatusToProgressionStep()` will always output garbage. This is the hardest class of bugs to find at runtime — no errors, just missing data.

### Gotchas

1. **`chain_of_thought_update`, `ask_user_question`, `user_answer` don't exist in OfficeAgent's MessageType enum on main.** They're only on the uncommitted `feat/PL-W001` branch. PRs mirroring these types must explicitly note this dependency and coordinate merge order.
   (source: `modules/message-protocol/src/types/message-type.ts` — grep confirms no matches)

2. **Source line references can be stale or fabricated.** The PR claimed `Source: core.ts:297-314` for FileInfo but the actual interface at those lines has completely different fields. Always verify `// Source:` references before trusting them — they're commentary, not enforced.

3. **Module-level `Map` in Nitro server functions is single-worker only.** `transportRegistry.ts` uses `const activeTransports = new Map<>()` for session-scoped state. This works for single-server dev but won't survive multi-worker deployment.
   (source: PR file `server/transportRegistry.ts`)

4. **`requestId` tracking gap in useCoworkStream.ts.** The hook sets `requestId: ''` on session init and never updates it. When `sendAnswer` is called, it passes empty `requestId`, which will likely cause server-side rejection. This is a functional bug.
   (source: PR file `hooks/useCoworkStream.ts:52`)

5. **office-bohemia repo ID for ADO API calls:** `74031860-e0cd-45a1-913f-10bbf3f82555` — different from OfficeAgent's `61458d25-9f75-41c3-be29-e63727145257`. Always verify repo ID when posting cross-repo PR comments.
