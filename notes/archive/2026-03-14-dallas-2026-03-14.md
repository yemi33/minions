# Dallas Learnings — 2026-03-14 (W023)

## Task
Read and summarize a Loop page from a SharePoint content storage URL.

## Findings

### Loop Page Access from SharePoint Content Storage URLs
- SharePoint content storage URLs (`/contentstorage/...`) require special handling — they are NOT the user's Copilot workspace (source: URL path `/contentstorage/x8FNO-xtskuCRX2_fMTHLRo8YaOBF1tMsxtMIJmILxo`)
- The `decode_url` MCP tool does NOT support SharePoint `:fl:` URLs — only Loop `/p/` and `/join/` URL formats (source: `mcp__loop__decode_url` error: "Expected Loop URL path like /p/{payload}")
- WebFetch returns 401 for SharePoint URLs (authenticated content) (source: WebFetch 401 error)
- Graph API returns 403 "accessDenied" for content storage sites (source: Graph API `/v1.0/sites/` 403 response)
- SharePoint REST API returns 401 "invalidScope" — Azure CLI app lacks `user_impersonation` for SPO (source: SharePoint API 401 response)

### Correct Workflow for SharePoint Loop URLs
1. **Decode the nav parameter** — it's base64-encoded, containing URL params: `s` (site path), `d` (driveId), `f` (itemId), `x` (workspace JSON with `w` and `i` fields)
2. **Extract workspace podId** from `x.w` (base64-encoded `ODSP|domain|driveId|itemId` string) — use this as `workspaceId` for Loop MCP
3. **List pages** via `mcp__loop__list_pages` with the workspace podId to find the target page and get correct `odspMetadata`
4. **Construct pageId** from listing's `odspMetadata` — NOT from the URL's nav params (driveId may differ slightly!)
5. **Read page** via `mcp__loop__get_page` with constructed pageId

### Critical Gotcha: DriveId Mismatch
- The driveId in the URL's `d` nav param (`...MH9F...`) differed from the actual driveId in the workspace listing (`...MG9F...`) (source: nav param vs `mcp__loop__list_pages` odspMetadata comparison)
- Always use the driveId from `list_pages` odspMetadata, NOT from the URL's nav parameter
- This single character difference (`H` vs `G`) causes 404 "itemNotFound" errors

### Platform Notes
- `python3` unavailable on this Windows machine (Microsoft Store stub) — use `node -e` for all scripting (confirmed from squad notes)
- `uuidgen` unavailable — use `require('crypto').randomUUID()` in Node.js

## Conventions
- Loop MCP `list_pages` can return very large results (280K+ chars for workspaces with many pages) — search by itemId programmatically rather than reading raw output
- The `pageId` field in `list_pages` results can be `undefined` — construct it manually from `odspMetadata` fields: `base64(domain + "," + driveId + "," + itemId)`
